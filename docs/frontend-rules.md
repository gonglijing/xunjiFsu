# 前端分析与开发约束

本文档面向 FSU 当前前端实现，目标不是描述理想化架构，而是为现有 `SolidJS + Vite` 管理台建立可执行、可评审、可持续收敛的开发约束。

## 1. 当前前端现状

前端目录集中在 `ui/frontend/src/`，当前主要分为：

- `pages/`：页面入口
- `sections/`：页面主体块
- `components/`：可复用组件
- `api/`：后端接口封装
- `utils/`：路由、鉴权、时间、错误等工具

当前实现已经具备以下基础：

- API 层已有统一鉴权与错误包装处理，入口见 `ui/frontend/src/api.js`
- 页面基本遵循“`pages` 包装 `sections`”的组织方式
- 路由、toast、抽屉、表格等已有基础实现

但当前也存在几个明确问题：

- 路由仍集中写在 `ui/frontend/src/App.jsx`，页面增长会持续放大维护成本
- 大型页面把“列表加载、轮询、表单、弹窗、详情抽屉、执行命令”堆在同一文件
- 内联样式大量存在，布局与视觉规则分散在页面 JSX 中
- 页面层仍在处理原始接口兼容逻辑，API 层抽象边界不够硬
- 原生 `confirm()`、局部 `setInterval()`、页面自定义错误处理等交互规则不统一
- 测试覆盖偏薄，复杂页面逻辑变更缺少约束

## 2. 总体目标

前端后续演进遵循三个目标：

1. 页面职责单一，避免单文件继续膨胀。
2. API、交互、样式三类规则统一，避免同类问题在不同页面重复实现。
3. 新增功能优先沿用现有基础设施，不再扩散“页面自写一套”的模式。

## 3. 强制规则

以下规则默认为强制要求。没有特殊说明时，新代码必须遵守。

### 3.1 路由规则

- 不允许继续在 `ui/frontend/src/App.jsx` 中追加大段 `switch/case` 路由分发逻辑。
- 新页面必须通过集中路由表管理。
- 新页面默认支持懒加载；只有首页或极高频页面才允许保留同步导入。
- `App.jsx` 只负责壳层布局、鉴权壳和路由挂载，不再承载页面选择逻辑。

### 3.2 页面分层规则

- `pages/` 只负责页面级装配，不直接实现重业务逻辑。
- `sections/` 负责页面主体业务块。
- `components/` 只放通用组件，不放特定页面的临时拼装逻辑。
- `api/` 只负责数据请求和返回值归一化，不包含页面状态。
- `utils/` 只放纯工具或通用行为，不放领域状态机。

### 3.3 单文件规模规则

- 单个 `jsx/js` 文件超过 300 行时，必须评估拆分。
- 单个页面或 section 同时包含以下四类职责中的三类及以上时，必须拆分：
  - 列表加载
  - 表单编辑
  - 弹窗或抽屉
  - 轮询或定时刷新
- 复杂页面应拆成：
  - 页面装配
  - 数据加载逻辑
  - 列表视图
  - 表单视图
  - 详情视图

### 3.4 API 使用规则

- 页面与组件不允许直接使用 `fetch()`。
- 所有后端访问必须通过 `ui/frontend/src/api/` 模块。
- API 模块必须返回“业务可直接消费”的数据结构，不允许把 `Array.isArray(res) ? res : res?.data || []` 这类兼容写法扩散到页面层。
- 页面不得处理 JWT、`401` 跳转、APIResponse 解包等通用逻辑。

### 3.5 数据获取规则

- 页面层不得散写 `loading/error/data` 三元组样板代码。
- 新增复杂页面必须抽出页面加载器或领域 hook，统一提供：
  - `data`
  - `loading`
  - `error`
  - `reload`
- 多接口并发加载必须在领域层完成，页面只消费结果。

### 3.6 轮询规则

- 所有轮询必须有明确常量名，禁止魔法数字直接写在 `setInterval()`。
- 轮询必须在 `onMount` 中启动，在 `onCleanup` 中销毁。
- 轮询必须可暂停；页面不可见、详情弹层打开或用户不在目标上下文时，应评估暂停。
- 同一页面不允许存在多个分散的匿名轮询逻辑。

### 3.7 错误处理规则

- 用户可见错误必须统一分层：
  - 页面首次加载失败：页面级错误提示
  - 提交失败：toast
  - 字段校验失败：表单内联错误
- 页面不得只 `console.error` 或只写裸字符串而不反馈给用户。
- 页面错误文案应统一走现有错误消息解析能力，不直接硬编码重复错误文本。

### 3.8 确认操作规则

- 禁止新增原生 `window.confirm()` / `confirm()`。
- 删除、清空、危险切换等动作必须使用统一确认弹层组件。
- 确认组件至少支持：
  - 标题
  - 描述
  - 确认按钮危险态
  - 提交中禁用

### 3.9 样式规则

- 禁止新增大段静态内联样式。
- 允许极少量动态样式，但仅限运行时必须计算的视觉值。
- 以下内容必须抽为 class 或通用样式：
  - 布局
  - 间距
  - 弹层尺寸
  - 表格容器高度
  - 卡片和状态区块
- 不允许继续在页面里堆叠 `display:flex; gap:...; padding:...` 这类重复布局片段。

### 3.10 视觉 token 规则

- 颜色、圆角、阴影、边框、间距统一走 CSS 变量或语义 class。
- 不允许在新代码里硬编码大量 `#hex`、`rgba(...)`、固定像素间距。
- 字体、字重、辅助色、状态色都应有统一来源，不允许页面私自定义一套。

### 3.11 基础组件复用规则

- 表格统一使用同一套基础表格抽象。
- 模态框、抽屉、空状态、加载态、状态 badge 不允许页面自己重新实现视觉结构。
- 对现有基础组件缺能力时，应先补基础组件，而不是页面复制一份。

### 3.12 表单规则

- 表单默认值必须通过工厂函数创建，不共享可变对象常量。
- 提交前必须经过显式 `normalize/serialize`。
- 提交中按钮必须禁用，避免重复提交。
- 提交成功后必须有统一成功反馈。
- 编辑态与新建态的差异必须在领域层明确，不允许在 JSX 内散落分支。

### 3.13 状态管理规则

- 业务数据状态、异步状态、UI 状态必须分层命名。
- 不允许使用含义模糊的通用状态名。
- 能从已有状态推导出的派生值应使用 accessor/computed，不要重复存储。

### 3.14 测试规则

- 新增复杂前端逻辑时，至少补一个测试。
- 优先测试以下内容：
  - API 归一化
  - 数据转换
  - 表单序列化
  - 纯逻辑工具
  - 页面关键交互
- 不能只改重逻辑页面而完全不补测试。

## 4. 推荐规则

以下规则不是硬性阻塞项，但默认推荐遵守。

- 页面级请求优先封装成 `loadXxx()` 或 `useXxxPage()` 风格
- 对高频轮询列表优先做增量合并，而不是每轮全量覆盖
- 页面中出现重复 API 组合调用时，及时下沉到领域层
- 组件 props 命名优先使用语义化命名，不使用 `data1/data2` 这类弱语义命名
- 可视化页面优先把布局 class 化，再讨论视觉优化

## 5. 针对当前代码的重点整改建议

当前最值得优先收敛的区域：

### 5.1 `App.jsx`

- 把手写 `switch` 路由改成集中式路由表
- 支持页面懒加载
- 让壳层只负责布局与登录态切换

### 5.2 `sections/Devices.jsx`

- 拆分为：
  - 列表数据层
  - 设备编辑弹窗
  - 写命令弹窗
  - 设备详情抽屉
- 把轮询和运行态合并逻辑抽为独立模块
- 替换原生确认框

### 5.3 `pages/Topology.jsx`

- 把大段内联布局样式迁移为 class
- 把资源列、网关列、北向列拆成独立展示组件
- 把详情加载逻辑下沉

### 5.4 Debug 页面

- `ModbusSerialDebugPage.jsx` 与 `ModbusTCPDebugPage.jsx` 结构高度相似，应抽共享表单块、帧展示块、结果表格块
- 把大量重复内联样式统一抽走

### 5.5 通用交互层

- 抽统一确认弹层
- 抽统一模态壳
- 抽统一表格加载/空态/错误态展示

## 6. 评审检查清单

前端改动提交前，至少检查以下问题：

- 是否新增了 `confirm()`？
- 是否新增了大段静态内联样式？
- 是否在页面里直接 `fetch()`？
- 是否把接口兼容逻辑写进页面？
- 是否让单文件继续膨胀？
- 是否重复实现已有基础组件能力？
- 是否补了必要测试？

如果以上任意问题答案为“是”，应优先整改后再合并。
